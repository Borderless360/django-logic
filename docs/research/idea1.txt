### Enhancing Monitoring and Fallback Mechanisms in Django-Logic for Improved Process Reliability

This technical requirements document formalizes a proposed solution to address critical visibility and performance issues in a Django-based system utilizing the django-logic library. The primary challenge stems from a lack of insight into execution problems: the system does not currently identify where bottlenecks occur, measure their impact, or quantify the execution time required for various tasks. This opacity particularly affects processes like order fulfillment, where it is unclear whether orders are genuinely stuck in the fulfilling stage or simply queued for processing. Without such metrics, troubleshooting and optimization efforts are reactive and inefficient.

The proposed enhancements introduce systematic measurement, logging, and automated handling mechanisms, including timeouts, retries, and fallbacks. These changes draw from prior experiences, such as suggestions from team members (e.g., Dima), and aim to create a more resilient system. By logging execution details and leveraging observability tools, the solution enables data-driven identification of slow functions. Periodic checks via Celery tasks will trigger interventions for exceeding thresholds, while configurable timeouts ensure flexibility without excessive resource use. This approach not only resolves immediate issues like timeouts with third-party couriers (e.g., Aramex and Canada Post) but also facilitates long-term optimization of the django-logic processes.

### Problem Statement

The current system exhibits several deficiencies:

- **Lack of Visibility**: There is no mechanism to pinpoint where problems are occurring, such as in specific functions or transitions.
- **Absence of Measurement**: Execution times for individual problems or tasks are not tracked, making it impossible to establish baselines or detect anomalies.
- **Uncertainty in Process States**: For instance, in order fulfillment, it is unclear if delays result from stuck processes (e.g., infinite loops) or queue backlogs.
- **Ineffective Handling**: Existing timeout and retry mechanisms operate in a suboptimal manner, without dynamic adjustments or comprehensive logging.
- **Optimization Gaps**: Without average execution data, functions that consistently run long cannot be proactively investigated or optimized.
- **Third-Party Integration Issues**: Timeouts or failures on external services (e.g., couriers) lead to unhandled stuck orders, impacting overall reliability.

These issues collectively result in delayed resolutions, potential data inconsistencies, and increased manual intervention.

### Proposed Solution Overview

To mitigate these problems, the solution incorporates the following high-level components:

- **Execution Time Logging**: Measure and log the performance of every function involved in transitions, side effects, callbacks, Celery tasks, or other django-logic elements.
- **Observability Integration**: Submit logs to tools like ClickHouse to compute averages and aggregates for function execution times.
- **Timeout and Fallback Triggering**: Implement a dynamic timeout system that uses calculated averages to detect overruns and invoke fallbacks.
- **Periodic Monitoring Task**: Deploy a Celery task running at regular intervals (e.g., every minute) to scan for long-running executions and initiate appropriate actions.
- **Retry Mechanisms**: For recoverable processes, enable automatic retries with delays.
- **Configuration Flexibility**: Allow adjustable timeout limits via low-overhead storage (e.g., Redis or Postgres) to fine-tune without runtime penalties.
- **Investigation Flagging**: For non-recoverable issues (e.g., infinite loops), record details in the database for manual review.

This solution modifies the django-logic library minimally to include logging capabilities, ensuring compatibility with existing workflows.

### Detailed Functional Requirements

### 1. Execution Time Measurement and Logging

- Every function within the django-logic framework must be instrumented to measure its execution time.
- This includes functions in transitions, side effects, callbacks, Celery tasks, and any other relevant components.
- Upon completion (or timeout), submit execution details to logs, including function name, start/end times, duration, and context (e.g., task ID, order ID).
- The django-logic library shall be updated to support this logging natively, without halting or altering core functionality.
- Logs must be formatted for easy ingestion by observability tools like ClickHouse.

### 2. Observability and Average Calculation

- Use logged data to calculate average execution times for each function via tools such as ClickHouse.
- Averages should be computed periodically (e.g., daily or on-demand) and stored accessibly for reference.
- Enable querying of these averages to identify functions that "run on average quite a long time."
- Once identified, these functions must be flagged for investigation, optimization, or problem-solving (e.g., why they are stuck).

### 3. Timeout Function and Fallback Triggering

- Develop a timeout function that references the calculated averages to determine thresholds.
- If a function exceeds its average execution time, trigger a fallback mechanism.
- Not all fallbacks will be effective; for example:
    - In simple transitions stuck in an infinite loop, options to kill the task are limited.
    - Do not terminate immediately; instead, create a database record indicating the function requires debugging and investigation, noting it runs above the average with no viable fallback.
- For more complicated processes, such as order fulfillment:
    - Leverage existing developed fallback functions.
    - Move the order to an "action required" state.
    - Attempt automatic re-fulfillment several times, with delays (e.g., in several minutes).
- This retry approach should resolve issues with third-party couriers like Aramex and Canada Post by allowing fulfillment despite timeouts on their side.

### 4. Periodic Monitoring via Celery Task

- Implement a Celery task that runs periodically (e.g., every minute).
- This task shall check currently executing tasks or functions.
- Identify those exceeding their average execution time.
- Upon detection, immediately trigger the appropriate fallback or record for investigation.

### 5. Timeout Limit Configuration

- Address the challenge of setting timeout limits without frequent database or Redis accesses.
- For most functions, transitions, and tasks, allow execution above initial limits; intervene only upon observed problems.
- Fine-tune problematic tasks/functions via:
    - A simple wrapper on the Python side.
    - Direct configuration on Celery tasks or functions.
- Limits may be adjustable dynamically:
    - Store configurations in Redis or Postgres (e.g., set a function to allow 30 minutes, adjustable to 40 or 10 minutes).
    - Ensure no overhead on execution-time accesses to the database; configurations are loaded once or cached.
- This setup enables system-wide fine-tuning to identify bottlenecks and optimize functions within Django-logic processes easily.

### Non-Functional Requirements

- **Performance**: Logging and monitoring must introduce minimal overhead; aim for <5% increase in execution time.
- **Scalability**: The solution should handle high volumes of tasks without degrading system performance.
- **Reliability**: Fallbacks and retries must be idempotent to avoid data corruption.
- **Security**: Database records for investigations must include audit trails (e.g., timestamps, user/context).
- **Maintainability**: Modifications to django-logic should be modular and well-documented.
- **Testing**: Require unit tests for logging, timeout logic, and Celery tasks; integration tests for fallback scenarios.

### Implementation Considerations

- **Library Modifications**: Update django-logic to include decorators or hooks for logging execution times.
- **Database Schema Additions**: Add tables/fields for investigation records (e.g., function_name, execution_time, average_time, timestamp, status).
- **Celery Integration**: Ensure the periodic task is scheduled via beat and handles concurrency safely.
- **Observability Setup**: Configure log forwarding to ClickHouse; define queries for averages (e.g., GROUP BY function_name).
- **Fallback Examples**:
    - Simple Transition: Log overrun and flag in DB.
    - Order Fulfillment: Invoke fallback, set to "action required," retry 3-5 times with 5-minute intervals.
- **Adjustment Mechanism**: Use a configuration model in Postgres or Redis keys (e.g., "function:order_fulfill:timeout=1800" seconds).

### Benefits and Expected Impact

- **Improved Visibility**: Provides a "table of execution" for every task, enabling quick bottleneck identification.
- **Enhanced Reliability**: Automates handling of timeouts, reducing stuck orders and manual interventions.
- **Optimization Opportunities**: Data-driven insights allow targeted optimizations, potentially reducing average execution times.
- **Third-Party Resilience**: Solves specific issues with couriers by enabling retries, fulfilling most orders despite external delays.
- **Flexibility**: Adjustable limits ensure the system can adapt to varying loads or requirements without code changes.

This comprehensive approach transforms the system from reactive to proactive, leveraging data to maintain high performance in Django-logic-driven processes. Implementation should proceed in phases: logging first, then monitoring, followed by timeouts and fallbacks, with thorough testing at each stage.
